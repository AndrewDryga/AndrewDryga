# Foundations: Observability, Cost, Reliability

Modern engineering orgs don’t fail for lack of features. They fail because
- No one can see what’s happening (observability),
- They can’t keep the thing alive (reliability),
- And when it does run, it costs too much (cost).

This post is the baseline playbook I apply to turn fragile systems into reliable, observable, cost‑sane platforms—while keeping teams fast. It’s intentionally boring: minimal tools, strong defaults, measurable outcomes.

> Principle
> - Start with the smallest loop that gives you feedback in hours, not quarters.
> - Improve signal quality, then reduce toil, then optimize costs.
> - Ship thin slices; measure impact; repeat.

---

## The Three Levers

### 1) Observability: See reality, quickly
- Outcomes over tools: focus on questions you can answer in 5 minutes during an incident.
- Start with RED + USE: Request Rate/Errors/Duration and Utilization/Saturation/Errors on critical resources.
- Prefer consistent, low‑cardinality metrics; logs as detail; traces as glue.

### 2) Reliability: Agree on what “good” means
- Reliability is a product requirement; encode it as Service Level Objectives (SLOs).
- Track 1–3 user‑facing SLOs per surface. Let them drive work and prioritization.

### 3) Cost: Make spend boring and predictable
- Costs should be visible, forecasted, and unitized (cost per request/tenant/feature).
- Budgets with alerts (not pages) and a monthly review with engineering + product.

---

## A Small, High‑Leverage Framework

Use this template per service/surface:

- Goal: e.g., “95% of homepage requests complete under 300ms over 30 days.”
- Signals: RED metrics; relevant infra metrics (CPU, memory, queue depth).
- Measures:
  - Metrics: `http_*`, `db_*`, queue length, worker lag
  - Traces: request path, major spans (web → service → db)
  - Logs: structured, only for decisions (errors, audits, business events)
- Levers:
  - Reliability: rollbacks, circuit breakers, backpressure, timeouts, retries
  - Observability: sampling, spans, exemplars, log volume policies
  - Cost: right‑size resources, autoscale bounds, storage TTL, data tiering

## Step 1 — Instrumentation (RED + basics)

If you ship only one change this week, ship this: consistent request metrics.

```/dev/null/example.ts#L1-120
// Minimal OpenTelemetry + metrics (Node/TS example).
// Instrument your HTTP entrypoint once. Keep names short and consistent.

import { context, trace } from "@opentelemetry/api";
import { MeterProvider, PeriodicExportingMetricReader } from "@opentelemetry/sdk-metrics";
import { OTLPMetricExporter } from "@opentelemetry/exporter-metrics-otlp-http";

const meterProvider = new MeterProvider();
meterProvider.addMetricReader(
  new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter({ url: process.env.OTEL_EXPORTER_OTLP_METRICS_ENDPOINT }),
    exportIntervalMillis: 15000,
  })
);
const meter = meterProvider.getMeter("web");

const reqCounter = meter.createCounter("http_requests_total", {
  description: "Total number of HTTP requests",
});
const errCounter = meter.createCounter("http_request_errors_total", {
  description: "Total number of HTTP request errors",
});
const reqDuration = meter.createHistogram("http_request_duration_ms", {
  description: "HTTP request duration in milliseconds",
  unit: "ms",
});

// In your request handler
export async function handleRequest(req: Request): Promise<Response> {
  const start = Date.now();
  const span = trace.getTracer("web").startSpan("http.request", undefined, context.active());

  let status = 200;
  try {
    // ... your request routing ...
    return new Response("ok", { status });
  } catch (e) {
    status = 500;
    errCounter.add(1, { route: "unknown", method: req.method, status: String(status) });
    return new Response("error", { status });
  } finally {
    const duration = Date.now() - start;
    const labels = { route: routeTag(req), method: req.method, status: String(status) };
    reqCounter.add(1, labels);
    reqDuration.record(duration, labels);
    span.end();
  }
}

function routeTag(req: Request): string {
  // Normalize dynamic segments to avoid high cardinality
  return req.url.replace(/\/\d+/g, "/:id").replace(/\/[0-9a-f-]{8,}/gi, "/:uuid");
}
```

Notes
- Normalize labels to avoid cardinality explosions.
- Export metrics on a predictable interval; traces can be head‑sampled at, say, 5–10% initially.
- Add exemplars on histograms if your backend supports it (linking traces to metrics buckets).

## Step 2 — SLOs (define “good”)

Keep SLOs few and user‑facing. Tie them to work selection.

```/dev/null/slo.yaml#L1-80
service: web
timeWindow: 30d

slos:
  - name: "Homepage latency"
    objective: 95
    indicator:
      type: latency
      source: http_request_duration_ms
      selector:
        route: "/"
        method: "GET"
    threshold: 300 # ms

  - name: "Signup success"
    objective: 99
    indicator:
      type: ratio
      good: http_requests_total{route="/signup",status=~"^2..$"}
      total: http_requests_total{route="/signup"}
```

Operationalize
- Burn‑rate alerts (fast + slow) instead of paging on every 500.
- Weekly reliability review: What burned error budget? What did we learn? What do we stop?

## Step 3 — Cost visibility (unit economics)

Pick one unit to start (per request/tenant). Plot over time.

```/dev/null/unit-cost.md#L1-120
# Unit cost model (sketch)

inputs:
  infra:
    - name: "edge-cdn"
      monthly_cost_usd: 200
    - name: "app-nodes"
      monthly_cost_usd: 1200
    - name: "db-cluster"
      monthly_cost_usd: 900
  business:
    requests_per_month: 5_000_000

formula:
  cost_per_request = sum(infra.monthly_cost_usd) / requests_per_month
  # Iterate: refine with storage, background jobs, per-tenant isolation as needed.

review:
  - Track weekly. Look for step changes, not noise.
  - Add alerts for +20% WoW deltas (not pages).
```

Add budgets + alerts in your cloud (pick one). Keep it simple:
- A monthly budget per environment with 80% and 100% thresholds.
- A spike detector (day‑over‑day anomaly) routed to a Slack channel, not pager.

## Step 4 — Make reliability boring

Tactics that pay rent:
- Timeouts and retries with jitter; never infinite retries.
- Backpressure: queue limits, shed load, 429 instead of cascading 500s.
- Circuit breakers around flaky dependencies.
- Rollback policy: “If SLO breaches or error budget burns fast, roll back first, then investigate.”
- Safe deploys: canary + automatic rollback when golden metrics regress.

## 30 / 60 / 90 — A pragmatic rollout

- 30 days
  - Ship RED metrics on all entrypoints.
  - Define 1 SLO per surface; wire burn‑rate alerts (no noisy pages).
  - Add a unit cost line (one number) to your weekly review.
- 60 days
  - Tracing across the critical path (web → service → db).
  - Backpressure + circuit breakers on top 1 external dependency.
  - Autoscaling with explicit min/max and alarmed cost deltas.
- 90 days
  - 2–3 SLOs per surface (latency + success + one domain metric).
  - Forecasting: quarter‑ahead cost and capacity with assumptions.
  - Replace toil with guardrails (pre‑prod checks, deploy health gates).

---

## Anti‑patterns to avoid

- Logging everything “just in case”: you’ll drown signal and pay for it.
- 25 SLOs no one reads: you need few, high‑leverage constraints.
- Big‑bang “observability platform” migrations: start small; centralize later.
- Treating cost as finance’s problem: if engineers don’t see it, it will drift.

---

## Minimal tech, strong defaults

You do not need an exotic stack to do this well:
- Metrics: Prometheus‑style or vendor‑equivalent. Histograms for latency.
- Traces: OpenTelemetry. Start with head sampling at 5–10%.
- Logs: structured JSON with retention and fields that answer questions.
- Dashboards: one per surface, one per dependency. No art projects.
- Reviews: one weekly 30‑minute session for reliability + cost.
