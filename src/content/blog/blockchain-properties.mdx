Right now we are building [a large project in a healthcare field](https://medium.com/nebo-15/national-health-service-on-elixir-and-kubernetes-b1c2551f0dbb) for our government. One of the most interesting problems is making sure it is cryptographically impossible to alter records in the database—even if someone compromises the infrastructure.

Why does that matter? Investigations must be able to rely on the authenticity of medical records, and the government reimburses providers based on this data. Tampering equates to fraud and, more importantly, patient safety risks.

> This post was originally written as an RFC. The design is now battle-tested, and I’ve cleaned it up for clarity and grammar while preserving the intent.

## Why not just run a blockchain?

The obvious answer in 2017 was “just use a blockchain.” In practice we had a few hard constraints:

- The database stores extremely sensitive data (national healthcare). We cannot replicate it beyond the trust boundary.
- Encryption alone is not enough. Algorithms age, bugs appear, and quantum attacks are no longer theoretical.
- Proof-of-work/proof-of-stake style consensus burns CPU we simply don’t have in the budget.

So we asked a better question: _how do we give the platform blockchain-like guarantees while keeping a centralized database?_ The answer hinges on strong signing practices plus tamper-evident storage.

## Data model (simplified)

Everything a physician submits is signed with their private key, issued by a certified authority. Rows are immutable; we only append new versions.

| Column                    | Type   | Modifiers |
| ------------------------- | ------ | --------- |
| `master_patient_index_id` | UUID   | not null  |
| `patient_first_name`      | string | not null  |
| `patient_last_name`       | string | not null  |
| `patient_passport_number` | string | not null  |
| `physician_personnel_id`  | UUID   | not null  |
| `previous_block_hash`     | string | not null  |
| `physician_signature`     | bytes  | not null  |

The `physician_signature` covers the entire row payload. `previous_block_hash` links each record to the Merkle root of the prior block.

Blocks are rolled every few minutes. The trade-off is simple: shorter windows mean lower exposure; longer windows reduce coordination overhead.

## Failure scenarios

### Transactions not yet in a block

To alter a pending transaction you’d need the physician’s private key. Deleting a pending row is detectable via audit logs. This mirrors how Bitcoin treats unconfirmed transactions.

### Transactions in the latest block

Every new write references the latest block hash. To tamper you would need either to:

- Delete the block entirely (which we detect), or
- Re-sign every row with each physician’s key (functionally impossible).

### Time is our ally

As more blocks confirm on top of a transaction, confidence increases—exactly like cumulative proof-of-work. We publish block headers externally so any third party can detect gaps.

## Implementation options we considered

1. A database with first-class immutability guarantees (none fit our needs in 2017).
2. A native extension enforcing signatures and hash chaining (risk: portability).
3. An Ecto adapter encapsulating the logic (works, but ties us to Elixir).
4. A dedicated Elixir service that enforces the rules while writing to PostgreSQL (our choice).

We landed on option 4 for maintainability and rollout speed.

### Storage engines

Append-friendly, Dynamo-inspired stores (Cassandra, Riak) are a great fit. They shard naturally and make partial rebuilds easy. When we eventually need to scale horizontally, this is where we’ll go.

### Consistency knobs (`N`, `W`, `R`)

We still need to balance consistency and availability:

- Writes that mutate business state use `W = ALL`.
- Read paths that materialize blocks use `R = ALL`.
- Non-critical reads (analytics, reporting) can be eventually consistent.

The beauty of tunable consistency is that we pick what matters per operation.

### Integrity checks

Periodically we recompute hashes and signatures:

- Chain verification: walk backwards, recomputing each hash.
- Batch verification: recompute the Merkle root for the block.
- Signature verification: validate with stored public keys.
- Snapshot verification: recompute state digests and check pointers.

Failures trigger an automated restore-from-backup, followed by another validation pass. If the check fails twice, the system halts and alerts humans.

## CAP refresher

| Choice | Guarantees         | Trade-off                                  |
| ------ | ------------------ | ------------------------------------------ |
| CP     | Strong consistency | You lose availability during partitions    |
| AP     | High availability  | Clients may read stale data                |
| CA     | Mythical           | Partitions _will_ happen in real systems   |

We deliberately choose CP for writes that affect money or medical records. Everything else can degrade gracefully.

## Open questions (still true years later)

- What is the runbook when integrity checks fail?
- How do we reject transactions that link to stale blocks without burning operators?
- Should we anchor block headers in a public ledger (e.g., Ethereum) for extra assurance?

These are policy decisions as much as technical ones.

## Takeaways

We achieved blockchain-style tamper evidence without adopting a full decentralized stack:

- Physicians sign everything they submit.
- Hash chaining + Merkle roots make tampering obvious.
- External anchors raise the bar for attackers.

The result is boring, audit-friendly infrastructure that satisfies regulators and keeps operations lean. That’s the goal.
